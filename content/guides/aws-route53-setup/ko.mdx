# AWS Route 53 설정하기

AWS Route 53은 확장 가능하고 가용성이 높은 클라우드 DNS(Domain Name System) 웹 서비스입니다. 이 가이드에서는 호스팅 영역을 생성하고 DNS 레코드를 설정하는 방법을 안내합니다.

<FreeTierInfo
  limit="호스팅 영역당 $0.50/월"
  period="계속"
  conditions={[
    "호스팅 영역: $0.50/월 (프리티어 없음)",
    "AWS 리소스 Alias 쿼리: 무료",
    "표준 쿼리: 처음 10억 개/월 $0.40"
  ]}
/>

<Callout type="info">
Route 53은 프리 티어가 없지만, 비용이 매우 저렴합니다. 호스팅 영역 하나당 월 $0.50이며, AWS 리소스(CloudFront, S3, ALB 등)에 대한 Alias 레코드 쿼리는 무료입니다.
</Callout>

## 사전 준비사항

- AWS 계정 ([aws-account-setup](/guides/aws-account-setup) 참고)
- 등록된 도메인 (Route 53 또는 외부 레지스트라)
- 선택사항: CloudFront 배포 또는 S3 웹사이트

---

<Step number={1} title="Route 53 대시보드 접속 및 호스팅 영역 생성">

AWS 콘솔에서 Route 53 서비스로 이동하여 호스팅 영역을 생성합니다.

1. [AWS Console](https://console.aws.amazon.com/)에 로그인합니다
2. 상단 검색창에 "Route 53"을 입력하고 선택합니다
3. 왼쪽 메뉴에서 **호스팅 영역**을 클릭합니다
4. **호스팅 영역 생성** 버튼을 클릭합니다

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step1-create-hosted-zone-ko.png"
  alt="호스팅 영역 생성 버튼"
  caption="호스팅 영역 생성 버튼을 클릭하여 새 DNS 호스팅 영역을 시작합니다"
/>

### 호스팅 영역 설정

5. **도메인 이름**: 소유한 도메인 입력 (예: `example.com`)
6. **설명**: 선택사항 (예: `My website DNS`)
7. **유형**: **퍼블릭 호스팅 영역** 선택 (인터넷에서 접근 가능)
   - 퍼블릭: 인터넷에서 접근 가능한 도메인
   - 프라이빗: VPC 내부에서만 사용하는 도메인

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step1-configure-zone-ko.png"
  alt="호스팅 영역 구성"
  caption="도메인 이름과 유형을 설정합니다"
/>

8. **태그**: 선택사항으로 태그 추가 가능
9. **호스팅 영역 생성** 버튼을 클릭합니다

### 네임서버 확인

10. 생성된 호스팅 영역이 표시됩니다
11. 호스팅 영역을 클릭하여 세부 정보를 확인합니다
12. **네임서버(NS)** 레코드에서 4개의 네임서버를 확인합니다:
    - `ns-1234.awsdns-12.org`
    - `ns-5678.awsdns-34.co.uk`
    - `ns-9012.awsdns-56.com`
    - `ns-3456.awsdns-78.net`

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step1-nameservers-ko.png"
  alt="네임서버 확인"
  caption="Route 53에서 제공하는 4개의 네임서버를 확인합니다"
/>

<Callout type="warning">
이 네임서버 주소는 도메인 레지스트라에 설정해야 합니다. 네임서버를 변경하지 않으면 DNS 레코드가 작동하지 않습니다.
</Callout>

</Step>

<Step number={2} title="도메인 레지스트라에서 네임서버 변경">

도메인 레지스트라(GoDaddy, Namecheap, Gabia 등)에서 네임서버를 Route 53으로 변경합니다.

### Route 53에서 도메인을 등록한 경우

- 네임서버가 자동으로 설정되므로 이 단계를 건너뛰세요

### 외부 레지스트라에서 도메인을 등록한 경우

1. 도메인 레지스트라의 관리 콘솔에 로그인합니다
2. 도메인 관리 또는 DNS 설정 페이지로 이동합니다
3. **네임서버 변경** 또는 **Custom DNS** 옵션을 찾습니다
4. Route 53에서 확인한 4개의 네임서버를 입력합니다:

<CopyBlock language="text">
ns-1234.awsdns-12.org
ns-5678.awsdns-34.co.uk
ns-9012.awsdns-56.com
ns-3456.awsdns-78.net
</CopyBlock>

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step2-registrar-nameservers-ko.png"
  alt="레지스트라 네임서버 설정"
  caption="외부 레지스트라에서 Route 53 네임서버로 변경합니다"
/>

5. 변경 사항을 저장합니다

<Callout type="info">
네임서버 변경은 전 세계에 전파되는 데 최대 48시간이 걸릴 수 있습니다. 일반적으로 몇 시간 내에 완료되지만, 완전히 전파될 때까지 기다려야 합니다.
</Callout>

### 네임서버 전파 확인

6. 터미널에서 다음 명령어로 네임서버 전파를 확인합니다:

<CopyBlock language="bash">
nslookup -type=NS example.com
</CopyBlock>

또는:

<CopyBlock language="bash">
dig example.com NS
</CopyBlock>

7. Route 53 네임서버가 표시되면 전파가 완료된 것입니다

<Step number={3} title="A 레코드 및 CNAME 레코드 추가">

도메인을 IP 주소 또는 다른 도메인으로 연결하는 DNS 레코드를 추가합니다.

### A 레코드 추가 (IPv4 주소)

1. 호스팅 영역 세부 정보 페이지에서 **레코드 생성** 버튼을 클릭합니다
2. **레코드 이름**:
   - 루트 도메인(`example.com`)은 비워둡니다
   - 서브도메인(`www.example.com`)은 `www` 입력
3. **레코드 유형**: **A - IPv4 주소로 트래픽 라우팅** 선택
4. **값**: 서버의 IP 주소 입력 (예: `192.0.2.1`)
5. **TTL**: **300** 초 (5분, 기본값)
6. **라우팅 정책**: **단순 라우팅** 선택

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step3-a-record-ko.png"
  alt="A 레코드 생성"
  caption="도메인을 IP 주소로 연결하는 A 레코드를 추가합니다"
/>

7. **레코드 생성** 버튼을 클릭합니다

### CNAME 레코드 추가

CNAME 레코드는 도메인을 다른 도메인으로 연결합니다.

8. 다시 **레코드 생성** 버튼을 클릭합니다
9. **레코드 이름**: 서브도메인 입력 (예: `blog`)
10. **레코드 유형**: **CNAME - 이름을 다른 이름으로 라우팅** 선택
11. **값**: 대상 도메인 입력 (예: `myblog.wordpress.com`)
12. **TTL**: **300** 초
13. **레코드 생성** 버튼을 클릭합니다

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step3-cname-record-ko.png"
  alt="CNAME 레코드 생성"
  caption="서브도메인을 다른 도메인으로 연결하는 CNAME 레코드를 추가합니다"
/>

<Callout type="warning">
**중요한 제한사항:**
- CNAME 레코드는 루트 도메인(`example.com`)에 사용할 수 없습니다
- 루트 도메인에는 A 레코드 또는 Alias 레코드를 사용해야 합니다
- 같은 이름에 CNAME과 다른 레코드를 함께 사용할 수 없습니다
</Callout>

### MX 레코드 추가 (이메일용, 선택사항)

14. 이메일 서비스를 사용하는 경우 MX 레코드를 추가합니다:
    - **레코드 이름**: 비워둡니다 (루트 도메인)
    - **레코드 유형**: **MX - 메일을 메일 서버로 라우팅**
    - **값**: `10 mail.example.com` (우선순위 포함)
    - **TTL**: **300** 초

</Step>

<Step number={4} title="Alias 레코드로 AWS 리소스 연결">

Alias 레코드를 사용하여 CloudFront, S3, Load Balancer 등 AWS 리소스에 연결합니다.

### Alias 레코드란?

Alias 레코드는 Route 53의 특별한 레코드 유형으로, AWS 리소스에 대한 쿼리가 무료이며 루트 도메인에도 사용할 수 있습니다.

### CloudFront 배포에 Alias 레코드 추가

1. **레코드 생성** 버튼을 클릭합니다
2. **레코드 이름**:
   - 루트 도메인은 비워둡니다
   - 또는 서브도메인 입력 (예: `www`)
3. **레코드 유형**: **A - IPv4 주소로 트래픽 라우팅** 선택
4. **별칭**: **예** 활성화

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step4-alias-enable-ko.png"
  alt="Alias 활성화"
  caption="별칭 옵션을 활성화하여 AWS 리소스에 연결합니다"
/>

5. **트래픽 라우팅 대상**:
   - **CloudFront 배포에 대한 별칭** 선택
   - 드롭다운에서 CloudFront 배포 선택
   - 또는 배포 도메인 이름 입력 (예: `d111111abcdef8.cloudfront.net`)

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step4-alias-cloudfront-ko.png"
  alt="CloudFront Alias 레코드"
  caption="CloudFront 배포에 대한 Alias 레코드를 추가합니다"
/>

6. **레코드 생성** 버튼을 클릭합니다

### S3 웹사이트 엔드포인트에 Alias 레코드 추가

7. S3 정적 웹사이트 호스팅을 사용하는 경우:
   - **레코드 유형**: **A**
   - **별칭**: **예**
   - **트래픽 라우팅 대상**: **S3 웹사이트 엔드포인트에 대한 별칭** 선택
   - 리전과 버킷 선택

### www 서브도메인도 추가

8. `www.example.com`도 같은 리소스로 연결하려면:
   - 위 과정을 반복하되, 레코드 이름에 `www` 입력
   - 같은 AWS 리소스를 대상으로 선택

<DevTip title="Alias vs CNAME 차이점">
**Alias 레코드의 장점:**
- AWS 리소스에 대한 쿼리가 무료
- 루트 도메인(apex)에 사용 가능
- 자동으로 대상 리소스의 IP 변경을 추적
- Zone apex 쿼리에 더 빠른 응답

**CNAME 레코드:**
- 외부 도메인에 연결할 때 사용
- 루트 도메인에 사용 불가
- 쿼리 비용 발생
</DevTip>

<Step number={5} title="DNS 레코드 테스트 및 검증">

설정한 DNS 레코드가 올바르게 작동하는지 테스트하고 검증합니다.

### 레코드 목록 확인

1. 호스팅 영역 세부 정보에서 추가한 모든 레코드를 확인합니다:
   - **NS** 레코드: Route 53 네임서버 (자동 생성)
   - **SOA** 레코드: 호스팅 영역 정보 (자동 생성)
   - **A** 레코드 또는 **Alias**: 루트 도메인
   - **CNAME** 또는 **Alias**: www 서브도메인
   - 기타 추가한 레코드

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step5-record-list-ko.png"
  alt="DNS 레코드 목록"
  caption="호스팅 영역에 추가된 모든 DNS 레코드를 확인합니다"
/>

### DNS 조회 테스트

2. 터미널에서 A 레코드를 테스트합니다:

<CopyBlock language="bash">
nslookup example.com
</CopyBlock>

또는:

<CopyBlock language="bash">
dig example.com
</CopyBlock>

3. CNAME 레코드를 테스트합니다:

<CopyBlock language="bash">
nslookup www.example.com
</CopyBlock>

4. Alias 레코드를 테스트합니다 (CloudFront):

<CopyBlock language="bash">
dig example.com
</CopyBlock>

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1/guides/aws-route53-setup/step5-dns-test-ko.png"
  alt="DNS 조회 테스트"
  caption="nslookup 또는 dig 명령어로 DNS 레코드를 확인합니다"
/>

### 웹 브라우저 테스트

5. 웹 브라우저에서 도메인으로 접속합니다:
   - `https://example.com`
   - `https://www.example.com`

6. 올바른 콘텐츠가 표시되는지 확인합니다

### Route 53 헬스 체크 (선택사항)

7. 왼쪽 메뉴에서 **상태 확인**을 클릭합니다
8. **상태 확인 생성** 버튼을 클릭합니다
9. 다음 설정을 구성합니다:
   - **이름**: `example-com-health-check`
   - **모니터링할 항목**: **엔드포인트**
   - **프로토콜**: **HTTPS**
   - **도메인 이름**: `example.com`
   - **경로**: `/` (또는 특정 경로)

10. **다음** 버튼을 클릭하고 알림 설정 (SNS 주제)을 구성합니다
11. **상태 확인 생성** 버튼을 클릭합니다

<Callout type="info">
헬스 체크는 엔드포인트의 가용성을 모니터링하고, 문제 발생 시 SNS를 통해 알림을 보냅니다. 헬스 체크는 월 $0.50의 비용이 발생합니다.
</Callout>

### 트래픽 정책 (고급, 선택사항)

Route 53 트래픽 정책을 사용하면 복잡한 라우팅 규칙을 생성할 수 있습니다:

- **가중치 기반 라우팅**: A/B 테스트, 카나리 배포
- **지연 시간 기반 라우팅**: 가장 빠른 리전으로 라우팅
- **지리적 위치 라우팅**: 사용자 위치에 따라 라우팅
- **장애 조치 라우팅**: 기본/백업 리소스 설정

## Route 53 도메인 등록

Route 53에서 직접 도메인을 등록할 수도 있습니다:

1. Route 53 대시보드에서 **도메인 → 등록된 도메인**을 클릭합니다
2. **도메인 등록** 버튼을 클릭합니다
3. 원하는 도메인을 검색합니다
4. 사용 가능한 경우 장바구니에 추가하고 결제를 진행합니다
5. 도메인 등록 시 호스팅 영역이 자동으로 생성됩니다

**도메인 가격 (예시):**
- .com: $13/년

## 문제 해결

### DNS가 전파되지 않습니다

1. **네임서버 확인**: 레지스트라에서 네임서버가 올바르게 설정되었는지 확인
2. **전파 시간**: 최대 48시간 대기 (보통 몇 시간)
3. **캐시 클리어**: 로컬 DNS 캐시를 클리어합니다:
   ```bash
   # Windows
   ipconfig /flushdns

   # Mac
   sudo dscacheutil -flushcache

   # Linux
   sudo systemd-resolve --flush-caches
   ```

### Alias 레코드가 작동하지 않습니다

1. **대상 리소스**: CloudFront 배포 또는 S3 버킷이 존재하는지 확인
2. **리전**: 올바른 리전을 선택했는지 확인
3. **CloudFront CNAME**: CloudFront 배포의 대체 도메인 이름에 도메인이 추가되었는지 확인

### HTTPS 인증서 오류

1. **CloudFront 인증서**: CloudFront용 인증서는 us-east-1에서 발급
2. **도메인 검증**: ACM 인증서가 완전히 검증되었는지 확인
3. **대체 도메인**: CloudFront 배포 설정에서 대체 도메인 이름 확인

### 높은 DNS 쿼리 비용

1. **TTL 증가**: TTL을 늘려 쿼리 수 감소 (예: 300초 → 3600초)
2. **Alias 사용**: AWS 리소스에는 CNAME 대신 Alias 레코드 사용
3. **불필요한 레코드**: 사용하지 않는 레코드 삭제

<Callout type="tip">
Route 53의 Alias 레코드는 AWS 리소스에 연결할 때 항상 무료이므로, CloudFront, S3, ALB, API Gateway 등에는 CNAME 대신 Alias 레코드를 사용하세요.
</Callout>
