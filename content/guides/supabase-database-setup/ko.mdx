# Supabase 데이터베이스 설정 및 테이블 생성 가이드

Supabase는 PostgreSQL 데이터베이스를 기반으로 하며, GUI 테이블 에디터와 SQL 에디터를 모두 제공합니다. 이 가이드에서는 테이블을 생성하고, 데이터를 관리하며, Row Level Security(RLS)를 설정하여 안전하게 데이터베이스를 사용하는 방법을 알아봅니다.

  limit="500MB database, unlimited tables and rows"

  status="available"
/>

## 시작하기 전에

다음 항목들이 준비되어 있는지 확인하세요.

- [Supabase 계정 및 프로젝트 생성](../supabase-account-setup) 완료
- 기본적인 데이터베이스 개념 이해 (테이블, 컬럼, 행)
- SQL 기본 문법 이해 (선택사항, GUI로도 작업 가능)

<Callout type="info">
PostgreSQL의 모든 기능을 사용할 수 있으며, JSON, 배열, 전문 검색(Full-text search), PostGIS 등 고급 기능도 지원합니다.
</Callout>

<Step number={1} title="테이블 생성하기 (Table Editor)">

GUI를 사용하여 첫 번째 테이블을 생성해봅시다. 예제로 블로그 포스트를 관리하는 `posts` 테이블을 만들겠습니다.

1. Supabase 대시보드에서 **Table Editor**를 클릭합니다
2. **New table** 버튼을 클릭합니다
3. 테이블 정보를 입력합니다:

   **Name**: `posts` (테이블 이름, 복수형 권장)

   **Description**: "Blog posts table" (선택사항)

   **Enable Row Level Security (RLS)**: 체크 (보안을 위해 항상 활성화 권장)

4. 기본 컬럼이 자동으로 생성됩니다:
   - `id`: bigint, Primary Key, auto-increment
   - `created_at`: timestamptz, 기본값: now()

  src="https://res.cloudinary.com/demo/image/upload/v1234567890/guides/supabase-database-setup/step1-create-table-ko.png"
  alt="Supabase 테이블 생성"
  caption="Table Editor에서 새 테이블을 생성합니다"
/>

<DevTip title="테이블 이름 컨벤션">
**권장 규칙:**
- 소문자 사용 (users, posts)
- 복수형 사용 (user 대신 users)
- 언더스코어로 단어 구분 (blog_posts, user_profiles)
- 예약어 피하기 (table, user, order 등)

**이유:**
- PostgreSQL은 대소문자를 구분하지 않음 (소문자로 변환)
- 복수형은 여러 행을 담는 테이블의 의미를 명확히 함
- 일관된 명명 규칙으로 코드 가독성 향상
</DevTip>

5. **Save** 버튼을 클릭하여 테이블을 생성합니다

<Step number={2} title="컬럼 추가 및 제약조건 설정하기">

이제 블로그 포스트에 필요한 컬럼들을 추가해봅시다.

1. 생성된 `posts` 테이블을 클릭합니다
2. **New Column** 버튼을 클릭합니다
3. 다음 컬럼들을 하나씩 추가합니다:

### title 컬럼 (제목)
- **Name**: `title`
- **Type**: `text`
- **Default Value**: (비워둠)
- **Is Nullable**: 체크 해제 (필수 필드)
- **Is Unique**: 체크 해제
- **Save** 클릭

### content 컬럼 (본문)
- **Name**: `content`
- **Type**: `text`
- **Default Value**: (비워둠)
- **Is Nullable**: 체크 (선택 필드, 초안 저장 가능)
- **Save** 클릭

### author_id 컬럼 (작성자 ID)
- **Name**: `author_id`
- **Type**: `uuid`

- **Is Nullable**: 체크 해제 (필수 필드)

- **Is Nullable**: 체크 (게시 전에는 null)

<DevTip title="PostgreSQL 데이터 타입 선택 가이드">
**자주 사용하는 타입:**

**텍스트:**
- `text`: 길이 제한 없는 텍스트 (제목, 본문, 설명 등)
- `varchar(n)`: 길이 제한 있는 텍스트 (이메일, 사용자명 등)
- `char(n)`: 고정 길이 텍스트 (국가 코드, 우편번호 등)

**숫자:**
- `integer` / `int4`: 정수 (-2,147,483,648 ~ 2,147,483,647)
- `bigint` / `int8`: 큰 정수 (ID, 카운터 등)
- `numeric(p,s)`: 정확한 소수 (금액, 정밀 계산)
- `real` / `float8`: 부동소수점 (과학 계산, 좌표)

**날짜/시간:**
- `timestamptz`: 타임존 포함 타임스탬프 (권장)
- `timestamp`: 타임존 없는 타임스탬프
- `date`: 날짜만
- `time`: 시간만

**기타:**
- `boolean`: true/false
- `uuid`: 고유 식별자 (사용자 ID, 세션 등)
- `json` / `jsonb`: JSON 데이터 (jsonb 권장, 인덱싱 가능)
- `array`: 배열 (태그, 카테고리 목록 등)
</DevTip>

<CopyBlock language="sql">
{`UPDATE posts

  published = true,

포스트 통계 조회:

<CopyBlock language="sql">

<CopyBlock language="sql">

  posts.published_at,
  users.email as author_email,
  users.name as author_name
FROM posts
INNER JOIN users ON posts.author_id = users.id
WHERE posts.published = true
ORDER BY posts.published_at DESC;`}
</CopyBlock>

<Step number={5} title="Row Level Security (RLS) 설정하기">

RLS는 데이터베이스 레벨에서 데이터 접근을 제어하는 PostgreSQL의 강력한 보안 기능입니다.

### RLS가 필요한 이유

<Callout type="warning">
RLS 없이는 클라이언트에서 모든 데이터에 접근할 수 있습니다! anon 키를 사용하더라도 누구나 테이블의 모든 데이터를 읽거나 수정할 수 있습니다.
</Callout>

### RLS 정책 생성하기

1. **Table Editor**에서 `posts` 테이블을 선택합니다
2. 우측 상단의 **RLS** 토글이 활성화되어 있는지 확인합니다
3. **Add RLS policy** 버튼을 클릭합니다

#### 정책 1: 모든 사용자가 게시된 포스트 읽기 가능

1. **Policy name**: "Public posts are viewable by everyone"
2. **Allowed operation**: `SELECT`
3. **Policy definition**: `Custom`
4. **USING expression**에 다음 입력:

<CopyBlock language="sql">
{`published = true`}
</CopyBlock>

5. **Save policy** 클릭

<Screenshot
  src="https://res.cloudinary.com/demo/image/upload/v1234567890/guides/supabase-database-setup/step5-rls-policy-ko.png"
  alt="RLS 정책 생성"
  caption="게시된 포스트만 조회할 수 있는 정책을 생성합니다"
/>

#### 정책 2: 작성자만 자신의 포스트 수정 가능

1. **Add RLS policy** 다시 클릭
2. **Policy name**: "Users can update their own posts"
3. **Allowed operation**: `UPDATE`
4. **Policy definition**: `Custom`
5. **USING expression**:

<CopyBlock language="sql">
{`auth.uid() = author_id`}
</CopyBlock>

6. **WITH CHECK expression**: (동일하게 입력)

<CopyBlock language="sql">
{`auth.uid() = author_id`}
</CopyBlock>

7. **Save policy** 클릭

#### 정책 3: 인증된 사용자만 포스트 생성 가능

1. **Add RLS policy** 클릭
2. **Policy name**: "Authenticated users can insert posts"
3. **Allowed operation**: `INSERT`
4. **Policy definition**: `Custom`
5. **WITH CHECK expression**:

<CopyBlock language="sql">
{`auth.uid() = author_id`}
</CopyBlock>

6. **Save policy** 클릭

<DevTip title="RLS 정책 이해하기">
**USING vs WITH CHECK:**
- **USING**: 기존 행에 대한 조건 (SELECT, UPDATE, DELETE)
- **WITH CHECK**: 새로 생성/수정되는 행에 대한 조건 (INSERT, UPDATE)

**auth.uid() 함수:**
- 현재 로그인한 사용자의 UUID 반환
- 비로그인 사용자는 `NULL` 반환
- Supabase Auth와 자동 연동

**일반적인 RLS 패턴:**

**1. 공개 읽기, 인증 쓰기:**
```sql
-- SELECT: 제한 없음
true

-- INSERT: 인증된 사용자만
auth.uid() IS NOT NULL
```

**2. 소유자만 접근:**
```sql
-- 모든 작업
auth.uid() = user_id
```

**3. 역할 기반 접근:**
```sql
-- 관리자만 접근
auth.jwt() ->> 'role' = 'admin'
```

**4. 조직/팀 기반 접근:**
```sql
-- 같은 팀 멤버만 접근
EXISTS (
  SELECT 1 FROM team_members
  WHERE team_members.team_id = posts.team_id
  AND team_members.user_id = auth.uid()
)
```
</DevTip>

### RLS 정책 테스트하기

SQL Editor에서 정책을 테스트할 수 있습니다:

<CopyBlock language="sql">
{`-- 게시된 포스트만 조회되는지 확인
SELECT * FROM posts;

-- 특정 사용자로 인증된 것처럼 테스트
SET request.jwt.claim.sub = 'user-uuid-here';
SELECT * FROM posts WHERE author_id = 'user-uuid-here';`}
</CopyBlock>

<CopyBlock language="javascript">

<DevTip title="Supabase 쿼리 빌더 팁">
**체이닝 메서드:**
- `.select()`: 조회할 컬럼 선택
- `.eq()`: 같음 (equals)
- `.neq()`: 같지 않음 (not equals)
- `.gt()`, `.gte()`: 크다, 크거나 같다
- `.lt()`, `.lte()`: 작다, 작거나 같다
- `.like()`, `.ilike()`: 패턴 매칭 (ilike는 대소문자 무시)
- `.in()`: 배열에 포함
- `.is()`: NULL 체크
- `.order()`: 정렬
- `.limit()`: 개수 제한
- `.range()`: 범위 지정 (페이지네이션)

## 문제 해결

### RLS 정책 때문에 데이터를 조회할 수 없어요

**원인**: RLS가 활성화되어 있지만 정책이 없거나 조건에 맞지 않음

**해결 방법**:
1. Table Editor에서 RLS 정책 확인
2. 개발 중에는 임시로 "모든 작업 허용" 정책 추가:
   ```sql
   -- 임시 정책 (프로덕션에서는 사용 금지!)
   CREATE POLICY "Allow all during development"
   ON posts FOR ALL
   USING (true)
   WITH CHECK (true);
   ```
3. 완성 후 적절한 RLS 정책으로 교체

### 외래 키(Foreign Key) 제약조건을 추가하고 싶어요

**방법**:
SQL Editor에서 다음 쿼리 실행:

<CopyBlock language="sql">
{`ALTER TABLE posts
ADD CONSTRAINT posts_author_id_fkey
FOREIGN KEY (author_id)
REFERENCES auth.users(id)
ON DELETE CASCADE;`}
</CopyBlock>

### JSON 컬럼을 추가하고 싶어요

**방법**:
1. Table Editor에서 New Column 클릭
2. Type을 `jsonb` 선택 (json보다 성능이 좋음)
3. 기본값 예시: `'{}'` 또는 `'[]'`

**사용 예시**:
<CopyBlock language="javascript">
{`// JSON 데이터 삽입
const { data, error } = await supabase
  .from('posts')
  .insert({
    title: '포스트',
    metadata: {
      tags: ['tech', 'tutorial'],
      views: 0,
      featured: true
    }
  })

// JSON 필드 쿼리
const { data } = await supabase
  .from('posts')
  .select('*')
  .contains('metadata', { featured: true })`}
</CopyBlock>

### 전문 검색(Full-text Search)을 구현하고 싶어요

**방법**:
SQL Editor에서 전문 검색 컬럼 추가:

<CopyBlock language="sql">
{`-- tsvector 컬럼 추가
ALTER TABLE posts
ADD COLUMN fts tsvector
GENERATED ALWAYS AS (
  to_tsvector('english', coalesce(title, '') || ' ' || coalesce(content, ''))
) STORED;

-- 인덱스 생성
CREATE INDEX posts_fts_idx ON posts USING GIN(fts);`}
</CopyBlock>

JavaScript에서 검색:

<CopyBlock language="javascript">
{`const { data } = await supabase
  .from('posts')
  .select('*')
  .textSearch('fts', 'supabase & postgresql')`}
</CopyBlock>
